CHALLENGES.txt — Review Feature Implementation
================================================================================

==================
> controllers/reviewController.js (existing)
> Added: checkIfBooked middleware + Booking, catchAsync, AppError imports
> HOW IT WORKS:
  - A new middleware `checkIfBooked` is added before `createReviews`.
  - It extracts the tour ID from `req.params.tourId` (nested route) or `req.body.tour`.
  - It queries the Booking model: `Booking.findOne({ tour: tourId, user: req.user.id })`.
  - If no booking is found, it returns a 403 error: "You can only review a tour
    that you have booked."
  - It also checks if the user has already reviewed the tour using
    `Review.findOne({ tour, user })`.
  - If a review already exists, it returns a 400 error telling them to update instead.
  - This ensures both booking verification AND duplicate review prevention at the
    API level.
==================

==================
> routes/reviewRoutes.js (existing)
> Added: reviewController.checkIfBooked to the POST route middleware chain
> HOW IT WORKS:
  - The POST '/' route now runs this middleware chain:
    1. authController.restrictTo('user')   — only 'user' role allowed
    2. reviewController.setTourUserIds     — auto-sets tour and user IDs
    3. reviewController.checkIfBooked      — verifies booking exists & no duplicate
    4. reviewController.createReviews      — creates the review (factory handler)
  - The checkIfBooked middleware sits after setTourUserIds so that req.body.tour
    and req.body.user are guaranteed to be populated before the booking check runs.
==================

==================
> controllers/viewsController.js (existing)
> Added: Review model import, booking/review check in getTour, passes hasBooked
  & hasReviewed flags to template
> HOW IT WORKS:
  - When a logged-in user visits a tour detail page, the controller queries:
    * Booking.findOne({ tour, user }) — checks if they've booked → hasBooked
    * Review.findOne({ tour, user })  — checks if they've already reviewed → hasReviewed
  - Both boolean flags are passed into the Pug template so the view can
    conditionally render the review form, a "thanks" message, or nothing at all.
  - For non-logged-in users, both flags default to false so nothing extra shows.

Hide Booking Section & Prevent Duplicate Bookings
================================================================================

==================
> models/bookingModel.js (existing)
> Added: Unique compound index on { tour, user }
> HOW IT WORKS:
  - `bookingSchema.index({ tour: 1, user: 1 }, { unique: true })` ensures
    MongoDB will reject any attempt to create a second booking for the same
    user + tour combination.
  - This is the database-level enforcement — even if the UI is bypassed
    (e.g. via API call), the duplicate booking will be rejected with a
    MongoDB duplicate key error (E11000).
  - Works in tandem with the UI-level check for a seamless user experience.
==================

==================
> views/tour.pug (existing)
> Changed: CTA section conditionally hides booking button for booked users
> HOW IT WORKS:
  - Uses the `hasBooked` flag already computed in `viewsController.getTour`.
  - If `user && hasBooked` is true:
    * Heading changes to "You're going on this tour!"
    * Text changes to a congratulatory message.
    * Button becomes "View My Bookings" linking to `/my-tours`.
    * The `#book-tour` button is NOT rendered, so the Stripe checkout
      cannot be triggered from the UI.
  - If the user has NOT booked:
    * Shows the normal "What are you waiting for?" CTA with the
      "Book tour now!" button (or "Log in" for anonymous users).
  - This is purely a UI guard — the model-level unique index provides
    the real enforcement against duplicates.

Nested Booking Routes & Improved Tour Dates
================================================================================

==================
> models/tourModel.js (existing)
> Changed: startDates from [Date] to array of subdocuments
> HOW IT WORKS:
  - Old: `startDates: [Date]` — just an array of plain dates.
  - New: Each startDate is a subdocument with three fields:
    * `date` (Date, required) — the actual tour start date.
    * `participants` (Number, default: 0) — current number of bookings.
    * `soldOut` (Boolean, default: false) — true when fully booked.
  - Each date now acts as a "tour instance" that tracks availability.
  - NOTE: After this change, existing data must be re-imported so dates
    use the new `{ date, participants, soldOut }` format.
==================

==================
> models/bookingModel.js (existing)
> Changed: Added startDate field, updated unique index
> HOW IT WORKS:
  - New `startDate` field (Date, required) stores which specific tour date
    the user booked.
  - Unique index changed from `{ tour, user }` to `{ tour, user, startDate }`.
  - This allows a user to book the SAME tour on DIFFERENT dates, but prevents
    booking the same tour+date combination twice.
==================

==================
> controllers/bookingController.js (existing)
> Added: setTourUserIds + checkAvailability middlewares, AppError import
> HOW IT WORKS:
  - `setTourUserIds`: Auto-populates `req.body.tour` from `req.params.tourId`
    and `req.body.user` from `req.user.id` for nested route support.
  - `checkAvailability`: Runs BEFORE `createBooking`:
    1. Validates that `tour` and `startDate` are provided.
    2. Loads the tour and finds the matching date subdocument using
       `tour.startDates.find(d => d.date.toISOString() === ...)`.
    3. If the date doesn't exist on the tour → 400 error.
    4. If `dateObj.soldOut === true` → 400 error ("fully booked").
    5. Increments `dateObj.participants += 1`.
    6. If `participants >= maxGroupSize` → sets `soldOut = true`.
    7. Saves the tour with `{ validateModifiedOnly: true }` to avoid
       triggering unrelated validations.
    8. Calls `next()` to proceed to `createBooking`.
  - This ensures real-time availability tracking at the database level.
==================

==================
> routes/bookingRoutes.js (existing)
> Changed: Enabled mergeParams, added middlewares to POST chain
> HOW IT WORKS:
  - `express.Router({ mergeParams: true })` allows the router to receive
    `tourId` and `userId` from parent routes.
  - POST '/' chain is now:
    1. `setTourUserIds` — auto-set IDs from URL params
    2. `checkAvailability` — validate date + check capacity
    3. `createBooking` — factory handler creates the booking
==================

==================
> routes/tourRoutes.js (existing)
> Added: Nested booking route delegation
> HOW IT WORKS:
  - `router.use('/:tourId/bookings', bookingRouter)` delegates to the
    booking router, passing tourId via mergeParams.
  - GET /api/v1/tours/:tourId/bookings → lists all bookings for a tour.
  - POST /api/v1/tours/:tourId/bookings → creates a booking for the tour.
==================

> routes/userRoutes.js (existing)

> Added: GET /forgot-password and GET /reset-password/:token
> HOW IT WORKS:
  - `/forgot-password` → renders the email input form.
  - `/reset-password/:token` → renders the new password form.
  - Neither requires authentication (anonymous users need access).

==================
> controllers/authController.js (existing)
> Changed: Reset URL now points to frontend page
> HOW IT WORKS:
  - Old: resetURL = `.../api/v1/users/resetPassword/${resetToken}`
  - New: resetURL = `.../reset-password/${resetToken}`
  - The email now sends users to the frontend form instead of a raw
    API endpoint, providing a proper UI experience.
==================

==================
> public/js/login.js (existing)
> Added: forgotPassword + resetPassword functions
> HOW IT WORKS:
  - `forgotPassword(email)`: POST to `/api/v1/users/forgotPassword`.
    On success → shows "Password reset link sent to your email!" alert.
  - `resetPassword(password, passwordConfirm, token)`: PATCH to
    `/api/v1/users/resetPassword/${token}`. On success → shows alert
    and redirects to homepage (user is auto-logged in via JWT cookie).
==================

==================
> public/js/index.js (existing)
> Added: forgotPassword/resetPassword imports + form handlers
> HOW IT WORKS:
  - `.form--forgot-password` submit: reads email, calls forgotPassword().
    Button shows "Sending..." during the request.
  - `.form--reset-password` submit: reads passwords + token from
    data attribute, calls resetPassword(). Button shows "Resetting...".

==================
> utils/email.js (existing)
> Changed: Email transport now uses SendGrid in both dev and production
> HOW IT WORKS:
  - Old: Checked `NODE_ENV === 'production'` → only used SendGrid in prod.
    In development, all emails went to Mailtrap (sandbox) and never
    reached real inboxes.
  - New: Checks if `SENDGRID_PASSWORD` exists → uses SendGrid regardless
    of NODE_ENV. Falls back to Mailtrap only if SendGrid is not configured.
  - This means password reset emails now reach the user's actual inbox
    even during local development.
